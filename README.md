# Analytics Platform — Refactor v2.0

> **dbt is the single source of truth.**
> FastAPI is a pure read-only serving layer on top of DuckDB marts.

---

## Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                        DATA FLOW                                │
│                                                                 │
│  Raw sources                                                    │
│      ↓                                                          │
│  dbt (staging → intermediate → marts)                           │
│      ↓  writes once                                             │
│  analytics.duckdb  ←──── SINGLE SOURCE OF TRUTH                │
│      ↓  read-only                                               │
│  FastAPI (serving layer — zero ETL logic)                       │
│      ↓  JSON                                                    │
│  React dashboard                                                │
└─────────────────────────────────────────────────────────────────┘
```

### Key design decisions

| Decision | Reason |
|---|---|
| dbt = only writer to DuckDB | Prevents logic drift between pipeline and API |
| FastAPI = read-only | `read_only=True` on DuckDB connection prevents accidents |
| One router per mart | Clear 1:1 mapping between endpoint and data source |
| Pydantic schemas mirror mart columns | Zero mapping overhead at serialization |
| Rule-based AI fallback | Project works without an Anthropic API key |
| `useApiBatch` hook | Single loading state for the whole dashboard |

## Quick Start

### Prerequisites
- dbt project already built (`dbt build` generates `analytics.duckdb`)
- Python 3.11+
- Node 18+

### 1. Backend

```bash
# Install deps
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# Configure
cp .env.example .env
# Set DUCKDB_PATH to your analytics.duckdb location

# Run
uvicorn backend.main:app --reload --port 8000
```

### 2. Frontend

```bash
cd frontend
npm install
npm run dev
# → http://localhost:3000
```

### 3. (Optional) AI Insights

```bash
# Add to .env:
ANTHROPIC_API_KEY=sk-ant-...
# Restart the API — /insights will use Claude
```

## dbt Mart Contracts

The API expects these columns to exist in each mart.
If your mart uses different names, update the SELECT aliases in the relevant router.

### `marts.mart_monthly_growth`
| Column | Type | Description |
|---|---|---|
| `month` | DATE / VARCHAR | YYYY-MM |
| `mrr` | FLOAT | Monthly Recurring Revenue |
| `mrr_mom_pct` | FLOAT | MoM growth % |
| `active_customers` | INT | Active accounts |
| `new_customers` | INT | New this month |
| `churned_customers` | INT | Churned this month |
| `churn_rate` | FLOAT | % churned |
| `nrr` | FLOAT | Net Revenue Retention % |
| `new_mrr` | FLOAT | Revenue from new customers |
| `expansion_mrr` | FLOAT | Revenue from expansions |
| `contraction_mrr` | FLOAT | Revenue lost to downgrades |
| `churn_mrr` | FLOAT | Revenue lost to churn |
| `net_new_mrr` | FLOAT | Net MRR change |

### `marts.fact_revenue`
| Column | Type | Description |
|---|---|---|
| `month` | DATE / VARCHAR | YYYY-MM |
| `movement_type` | VARCHAR | new / expansion / contraction / churn |
| `revenue` | FLOAT | Revenue amount |

### `marts.dim_channel`
| Column | Type | Description |
|---|---|---|
| `channel` | VARCHAR | Channel name |
| `customers_acquired` | INT | Total customers from channel |
| `total_revenue` | FLOAT | Total revenue attributed |
| `churn_rate` | FLOAT | Channel churn rate |
| `roi` | FLOAT | ROI if cost data available |

### `marts.mart_customer_metrics`
| Column | Type | Description |
|---|---|---|
| `customer_id` | VARCHAR / INT | Unique ID |
| `customer_name` | VARCHAR | Display name |
| `plan` | VARCHAR | Pricing tier |
| `channel` | VARCHAR | Acquisition channel |
| `mrr` | FLOAT | Current MRR |
| `ltv` | FLOAT | Lifetime value |
| `months_active` | INT | Tenure in months |
| `status` | VARCHAR | active / churned |
| `health_score` | FLOAT | 0–100 health score |

### `marts.fct_marketing_funnel`
| Column | Type | Description |
|---|---|---|
| `period` | VARCHAR | YYYY-MM or YYYY-QN |
| `stage` | VARCHAR | visitor / lead / mql / sql / trial / paid |
| `stage_order` | INT | Sort order (1 = top of funnel) |
| `count` | INT | Users at this stage |

## Data Layer Separation

```
ingestion  (optional)    data/simulate.py, data/etl.py
                              ↓ only used to seed demo DB
transformation (dbt)     dbt build → analytics.duckdb
                              ↓ single writer
serving (FastAPI)        read-only queries on marts
                              ↓ JSON
presentation (React)     useApi hooks → components
```

`data/simulate.py` and `data/etl.py` are **isolated** from the core serving path.
They exist only to generate a demo `analytics.duckdb` when no real dbt source is available.
In production, dbt runs on a schedule (Airflow, dbt Cloud, etc.) and writes the DB.
